## API credentials

Request a unique `clientId` and `clientSecret` from YourCoach. This pair
uniquely identifies your company application and is mandatory for API handshake
processes.

## Security keys & JWT setup

The YourCoach widget requires a JSON Web Token (JWT, RFC 7519) to securely map
your active session user to a YourCoach session. YourCoach validates the
authenticity of this token by verifying its cryptographic signature:

<Steps>
  <Step title="Generate an RSA key pair">
    Generate an RSA private/public key pair on your infrastructure. See [key
    generation script](/scripts/key-generation).
  </Step>
  <Step title="Share the public key">
    Provide the generated public key to YourCoach. Keep the private key secure
    on your server.
  </Step>
  <Step title="Sign JWTs on each request">
    During user authorization, generate a signed JWT containing the user's ID
    using your private key. See [JWT generation
    script](/scripts/jwt-generation).
  </Step>
</Steps>

## Host page whitelisting

<Check>
  Provide YourCoach with every origin where the widget iFrame will be embedded.
</Check>

The host origin is validated against the widget's `frame-ancestors` policy on
each load. Submit one entry per environment you plan to integrate from:

- **Production** — every customer-facing origin that will embed the widget.
- **Staging / Pre-production** — every pre-prod origin used for QA.
- **Local development** _(optional)_ — your loopback origin (e.g.
  `http://localhost:1234`) if you need the widget to run against YourCoach
  during local integration work.

### Supported formats

| Pattern                     | Matches                                         | Example                   |
| --------------------------- | ----------------------------------------------- | ------------------------- |
| Exact origin                | Single host, scheme, and port                   | `https://app.example.com` |
| Wildcard subdomain          | Any single-label subdomain of the parent domain | `https://*.example.com`   |
| Loopback with explicit port | Local dev server bound to a specific port       | `http://localhost:1234`   |

Scheme (`http` vs `https`) and port are both part of the origin —
`http://localhost:3000` and `http://localhost:4000` are distinct entries and
must each be submitted if both are used.

to its shared S3 bucket (or supply SFTP credentials) for the following AWS Role
ARNs:
